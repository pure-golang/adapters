# Definition of Done (DOD) для Адаптеров

Этот документ определяет критерии готовности при добавлении новых адаптеров в проект `adapters`.

## Критерии готовности

### 1. Требования к коду

- [ ] Код соответствует конвенциям проекта (см. `AGENTS.md`, раздел Global Conventions)
- [ ] Структура `Config` с тегами `envconfig` — см. skill `env_config`
- [ ] Реализация интерфейса `Provider` (или `RunableProvider` для долгоиграющих сервисов) — см. `AGENTS.md`, раздел Core Interfaces
- [ ] Обработка ошибок с заворачиванием в контекст через `errors.Wrap()` — см. `AGENTS.md`, раздел Global Conventions
- [ ] Использование `context.Context` первым аргументом во всех операциях, которые могут блокироваться
- [ ] Реализация `defer` для корректной очистки ресурсов

### 2. Требования к документации

- [ ] Наличие `doc.go` с package-level комментарием — см. skill `doc_go`
- [ ] `doc.go` актуален и отражает текущий публичный API
- [ ] Наличие `README.md` в директории адаптера с примерами использования (на русском)
- [ ] Все комментарии и документация на русском языке

### 3. Требования к тестированию

#### Обязательные тесты

- [ ] **Unit-тесты** для основной бизнес-логики — см. skill `testing_conventions`
- [ ] **Интеграционные тесты** с `github.com/testcontainers/testcontainers-go` — см. skill `integration_testing`

#### Интеграционные тесты (обязательно)

- [ ] Поддержка флага `-short`: `t.Skip("integration test")`
- [ ] Автоматический запуск и остановка контейнеров через testcontainers
- [ ] Ожидание готовности сервиса перед выполнением тестов
- [ ] Корректная очистка ресурсов после завершения тестов

#### Покрытие тестами

Для адаптеров баз данных и очередей сообщений:

- [ ] Тест подключения и установления соединения
- [ ] Тест базовых операций (CRUD)
- [ ] Тест обработки ошибок (включая нарушение ограничений)
- [ ] Тест транзакций (commit/rollback) — если применимо
- [ ] Тест уровней изоляции транзакций — если применимо
- [ ] Тест обработки таймаутов и отмены контекста
- [ ] Тест конкурентных операций

### 4. Требования к наблюдаемости

Подробности — см. skill `observability`

- [ ] OpenTelemetry tracing для всех операций (трейсер как пакетная переменная, именование `packageName.operation`, стандартные атрибуты)
- [ ] Интеграция со структурированным логированием (`log/slog`)
- [ ] Метрики Prometheus — если применимо

### 5. Дополнительные требования

- [ ] `golangci-lint run` выдаёт "0 issues."
- [ ] `go build ./...` выполняется без ошибок
- [ ] `go test ./...` выполняется без ошибок
- [ ] `go vet ./...` не выдаёт предупреждений
- [ ] `go mod tidy` не изменяет `go.mod` и `go.sum`
- [ ] Отсутствие `TODO` в коде адаптера (или заменены на Issue)
- [ ] `AGENTS.md` обновлён при введении новых паттернов

## Чеклист для Pull Request

Перед созданием PR для нового адаптера:

1. [ ] Все пункты DOD выполнены
2. [ ] Тесты проходят локально (включая интеграционные при запущенном Docker) — см. skill `go_commands`
3. [ ] Тесты проходят в CI
4. [ ] `README.md` содержит полную документацию
5. [ ] `AGENTS.md` обновлён при необходимости
6. [ ] Код отформатирован (`go fmt ./...`)
7. [ ] Нет лишних зависимостей в `go.mod`
